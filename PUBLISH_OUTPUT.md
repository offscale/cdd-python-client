# Publishing & Automating the Generated Client

This document outlines how to publish the **generated API client** (the output of the `cdd sync` command) and how to keep it automatically synchronized with your upstream server's OpenAPI specification.

## 1. Packaging the Generated Client

When `cdd sync` produces a `client.py` and related models, you need to wrap them in a package before distributing. A standard minimal Python package structure looks like this:

```
my-api-client/
├── pyproject.toml
├── README.md
└── src/
    └── my_api_client/
        ├── __init__.py
        └── client.py      <-- Generated by CDD
```

### Example `pyproject.toml`
Create a `pyproject.toml` in your client repository:

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "my-api-client"
version = "1.0.0"
description = "Auto-generated OpenAPI client"
dependencies = [
    "pydantic>=2.0",
    "urllib3"
]

[tool.hatch.build.targets.wheel]
packages = ["src/my_api_client"]
```

## 2. Publishing the Client to PyPI

To manually publish the generated client to the Python Package Index:

```bash
cd my-api-client
pip install build twine
python -m build
python -m twine upload dist/*
```

## 3. Automating Updates via GitHub Actions

The power of CDD is ensuring your client SDK is never out of sync with your API server. You can use a GitHub Actions Cron Job to routinely check your upstream OpenAPI server, sync the models, and publish a new version.

### Setup the GitHub Action Workflow

Create a file in your client repository at `.github/workflows/sync-and-publish.yml`:

```yaml
name: Sync OpenAPI Client

on:
  schedule:
    # Run every day at midnight UTC
    - cron: '0 0 * * *'
  workflow_dispatch: # Allow manual trigger via GitHub UI

jobs:
  sync_and_publish:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Client Repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install cdd-python-client build twine bumpver

      - name: Fetch Latest OpenAPI Spec
        # Replace the URL with your server's live openapi.json endpoint
        run: curl -s https://api.my-service.com/openapi.json > openapi.json

      - name: Sync Client Code
        # This updates the `src/my_api_client` files based on the fetched JSON
        run: cdd sync --from-openapi openapi.json --to-python src/my_api_client/

      - name: Check for changes
        id: git-check
        run: |
          git diff --exit-code || echo "changes=true" >> $GITHUB_OUTPUT

      - name: Bump Version and Commit
        if: steps.git-check.outputs.changes == 'true'
        run: |
          git config --global user.name "github-actions[bot]"
          git config --global user.email "github-actions[bot]@users.noreply.github.com"
          
          # Automatically bump the patch version in pyproject.toml
          bumpver update --patch
          
          git add src/my_api_client/ openapi.json pyproject.toml
          git commit -m "chore(openapi): auto-sync client with upstream spec"
          git push

      - name: Build and Publish to PyPI
        if: steps.git-check.outputs.changes == 'true'
        env:
          TWINE_USERNAME: __token__
          TWINE_PASSWORD: ${{ secrets.PYPI_API_TOKEN }}
        run: |
          python -m build
          twine upload dist/*
```

### How it Works:
1. **Cron Schedule**: The action runs daily at midnight (or manually).
2. **Fetch**: It pulls the absolute latest `openapi.json` from the upstream server.
3. **CDD Sync**: It runs the `cdd sync` CLI tool to regenerate the client, mock, and test files using the new spec.
4. **Diff Check**: It uses `git diff` to see if the generated AST code differs from what is currently checked in.
5. **Bump & Commit**: If the spec changed, it bumps the version in `pyproject.toml` and commits the code.
6. **Publish**: It builds the new wheel and uploads it to PyPI automatically (requires a `PYPI_API_TOKEN` secret to be set in your GitHub repository settings).
